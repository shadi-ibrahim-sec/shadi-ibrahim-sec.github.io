Findings — click to expand

The order-creation endpoint POST /api/orders accepted price and total directly from the client JSON body without server-side re-validation. By intercepting the checkout request in Burp Suite and modifying "price": 79.99 to "price": 2, I received a 201 Created response — a fully confirmed order at an arbitrary price.

⚡ Impact: Any authenticated user can purchase any item for $0.01. Because the platform uses Cash on Delivery with no third-party payment gateway, there is no downstream re-validation. Uncapped financial loss at scale.

Navigating directly to /admin/orders while authenticated as a standard "role":"User" account rendered the full admin panel with zero server-side role check. Every customer's full name, phone number, delivery address, and order history was immediately visible. No special tooling required — just a URL.

⚡ Impact: Mass PII exposure for every registered customer. Broken access control requires no exploit, no escalation — just navigation. Classified Critical under OWASP A01:2025.

Seven consecutive failed login attempts against the same account returned identical, immediate "Invalid login credentials" responses — no lockout, delay, or CAPTCHA triggered. The endpoint is fully open to automated brute-force and credential-stuffing at scale.

⚡ Impact: Users with weak or previously-breached passwords are at elevated risk of account takeover via automated attacks.

Submitting "price": -10 caused an unhandled server exception returning HTTP 500 with empty body — no graceful validation or error handling. Confirms complete absence of numeric boundary validation on the order endpoint.

⚡ Impact: Unhandled crash condition. Could be leveraged as a building block for availability degradation. Indicates broader input validation gaps.

Four controls were actively tested and confirmed working: Stored XSS — React's output encoding correctly rendered the payload as an inert text node. CSRF — SameSite=Lax cookie attribute blocks cross-origin POST requests. Admin Write Authorization — order status update correctly returned 403 Forbidden for non-privileged accounts. IDOR — cross-account direct order ID access returned 404, session-scoped correctly.

✓ These controls are correctly implemented and should be preserved during remediation. The write/read authorization inconsistency identified above makes these results particularly valuable context.
